# ARCHITECTURE.md

System architecture for Ava. Read alongside `CLAUDE.md` and `docs/COMPLIANCE.md`.

## High-level

```
Browser (Next.js client)
   │
   ├─ Text chat ──────► /api/chat ──► Safeguard Pipeline ──► Claude Haiku ──► filtered response
   │
   ├─ Avatar session ─► /api/avatar/session ──► HeyGen LiveAvatar (WebRTC) ◄─► ElevenLabs Agent
   │                                              (STT → LLM → TTS → lip-sync render)
   │
   ├─ Profile/score ──► /api/profile ──► radar compute + report (Claude Sonnet)
   │
   └─ Brag card ──────► /api/share ──► templated card render + (optional) templated Ava clip
                                          │
                                  Supabase (Postgres, RLS, encrypted health fields)
                                          │
                                       Stripe (payments, webhooks)
```

## Two pipelines — keep them separate

### 1. Text chat pipeline (cheap, fully controlled)
```
User types
  → /api/chat
  → [Layer 1] Emergency detection (deterministic keyword scan; if hit → 988/911 card, BYPASS LLM)
  → [Layer 2] System prompt constitution (defines what Ava can/cannot say)
  → Claude Haiku
  → [Layer 3] Output filter (banned phrases + medical-claim drift; deterministic)
  → [Layer 4] Response validator (valid JSON, score bounds, required fields)
  → response to client
```
This is the ONLY chat path. There is no un-safeguarded call to Claude anywhere in the codebase.

### 2. Avatar pipeline (expensive, metered)
```
User starts avatar session (must be authenticated + have credits)
  → /api/avatar/session creates a HeyGen LiveAvatar session (Lite mode)
  → ElevenLabs Conversational Agent handles STT → LLM → TTS
  → HeyGen renders lip-synced video, streamed to browser via WebRTC
  → credit ledger decrements per minute of session wall-clock
  → on session end: Claude Sonnet generates summary + extracts facts → Supabase
```

**Metering rule:** bill the full session wall-clock, not avatar-speech time. The credit decrement runs on a server-side timer tied to the live session, not on a "spoke N words" basis. See `docs/DATA-MODEL.md` credit ledger.

**Concurrency rule:** Essential plan = 20 concurrent sessions. Before creating a session, check active-session count. If at cap → return a `text_fallback` response and route the user to the (cheaper) text pipeline with a friendly message. Never block silently.

## OPEN DECISION — avatar LLM routing (flag before building avatar conversation)

The live avatar conversation LLM is one of:
- **(a) ElevenLabs bundled LLM** — cost is inside the ~$0.08–0.12/min agent rate. Simpler. **This is the launch default.**
- **(b) Claude Sonnet routed separately** — better quality control + your own safeguard layer in the loop, but adds ~$0.01–0.03/min and integration complexity.

Do not implement (b) without explicitly confirming — it changes unit economics. If quality testing in Phase 2 shows (a) is insufficient for the avatar experience, raise the switch as a deliberate decision with cost impact noted.

> Note: even with (a), the avatar agent's system prompt must carry the same constitution as the text pipeline, and a post-session transcript pass through the output filter is required for compliance logging.

## Profile computation

- The radar score (0–100 per axis + overall) is computed **deterministically** from structured signals Claude extracts from the conversation — NOT by asking the LLM "what's his score." The LLM extracts symptom signals into a typed schema; a pure function maps signals → scores. This keeps scoring auditable and prevents the model from inventing numbers.
- Scoring tiers and labels are fixed (see `docs/PRODUCT-SPEC.md`). Labels are deliberately non-diagnostic.

## Two profile artifacts

- **Private profile:** full radar + written report (Claude Sonnet) + clinician questions. Stored in-account, RLS-protected, never exposed to share endpoints.
- **Brag card:** generated by `/api/share` from a *whitelisted* subset of fields (overall score, progress delta, radar silhouette geometry). The share endpoint physically cannot read sensitive fields — enforce this with a separate DB view/RLS policy, not just application logic. See `docs/DATA-MODEL.md` §Share view.
