## What we're building

The **Decision CTA**: the post-Gap monetization beat in the conversation flow. After the Gap reveal teases the full profile, an inline **Decision card** offers the **$29 Wellness Profile** — and buying it (via real **Stripe test-mode** checkout) delivers the **written wellness-profile report** on a dedicated, in-account `/profile` page.

This is the first real monetization surface in the product. It is built carefully because it is the front edge of the entire 1D credit-liability subsystem.

### North-star fit

The Decision converts the exact promise the Gap makes ("your full profile fills in all six — with a written read"). It deepens the relationship by giving the user a durable, returnable private artifact (the `/profile` page — the future home of trends/history → the subscription hook). "Continue free" is always one tap away; there is no manufactured urgency, no blocking, no fake checkout. Value-earned conversion only (CLAUDE.md §Product north star).

### Scope decisions (settled in brainstorming)

- **Real test-mode purchase**, not an intent-capture shell. Stripe-hosted checkout, `sk_test_`/`pk_test_`. No card data touches our servers. Live mode (`sk_live`) is GATE-1-gated and out of scope.
- **Single hero product:** $29 **Wellness Profile** only. Starter $9 and Deep $49 are deferred (no distinct deliverable yet). No price-anchoring with unfulfillable tiers (compliance: deliver what you sell).
- **Deliverable = the written report** (the private profile artifact), generated by the existing Sonnet summarizer infrastructure, shown on a dedicated `/profile` page.
- **Decision card placement:** inline in the chat thread (same pattern as the Gap card), firing **~2 user-turns after the Gap**.

## Behavior

### The Decision card (in-chat)

- Fires **once per session**, on the first user turn that lands **≥ `DECISION_DELAY_TURNS` (2)** user-turns after the Gap card was shown. Deterministic, client-side — same philosophy as the 1D-a Gap trigger (not model-driven).
- **Suppressed** when the user already holds a `paid`/`ready` Wellness Profile entitlement (server passes an `alreadyPurchased` flag on initial load). Reset on `endCheckIn` alongside `gapShown`.
- Renders (reusing `AvaBubble`): the $29 Wellness Profile pitch ("All six axes + a written read on what they mean together"), a primary **"Unlock my profile — $29"** button, a quiet **"Continue free"** dismiss, and a small optional **"Share instead"** referral link (reuses the existing brag-card/`ShareBaseline` path — referrals stay always-optional per spec).
- **"Continue free"** sets a local `decisionDismissed` and the conversation continues untouched. No blocking, no re-prompt.

### Buy flow

1. Tap "Unlock my profile" → `createCheckout()` (never-throws browser wrapper) → `POST /api/checkout` → `{ url }` → `window.location = url` (Stripe-hosted page).
2. Pay in the Stripe sandbox → redirect to `/profile?checkout=success`.
3. `/profile` fires `POST /api/profile/generate` with the still-held transcript → "preparing…" → renders the report once `ready`.
4. Cancel → `/home?checkout=cancelled`; chat resumes; the card shows a gentle "whenever you're ready" state.

### The `/profile` page (private artifact)

Authed, RLS owner-only. States:
- `status='ready'` → full radar (all six) + the written read (decrypted server-side).
- `status='paid'`, report still generating → "preparing your profile…" (triggers/polls generation).
- No entitlement → locked state with the $29 upsell (route is safe to visit directly).

The report is the **private profile** (CLAUDE.md rule #5): output-filtered (non-diagnostic), encrypted at rest, in-account only, structurally unable to reach the brag card / share endpoint.

## Architecture

### Data model — migration `0007_credits_profiles`

**`credit_ledger`** — the liability spine (`CREDIT-LIABILITY-AND-EXIT.md`):

| column | type | purpose |
|---|---|---|
| `id` | uuid PK default gen_random_uuid() | |
| `user_id` | uuid NOT NULL FK → users(id) cascade | |
| `delta` | integer NOT NULL | + grant, − consume/expire/refund (whole credits) |
| `reason` | text NOT NULL | e.g. `purchase:wellness_profile` |
| `unit_price_cents` | integer NOT NULL default 0 | cash paid per credit this grant (refund/liability math) |
| `expires_at` | timestamptz | grants only; `now() + CREDIT_EXPIRY_MONTHS` (12); null otherwise |
| `stripe_event_id` | text UNIQUE | idempotency — webhook retry → `23505` no-op |
| `created_at` | timestamptz NOT NULL default now() | |

`balance = sum(delta)`. **RLS:** owner-only `SELECT` (`auth.uid() = user_id`); **no user INSERT/UPDATE/DELETE** — grants are written only by the service-role client in the webhook. (A user reads their balance; never mints credits.)

**`wellness_profiles`** — paid private-profile artifact + entitlement:

| column | type | purpose |
|---|---|---|
| `id` | uuid PK | |
| `user_id` | uuid NOT NULL FK → users(id) cascade | |
| `session_id` | uuid | which check-in the report reads |
| `stripe_checkout_id` | text UNIQUE | ties the unlock to the payment; idempotent upsert |
| `status` | text NOT NULL | `paid` → `ready` |
| `report` | text | the written read, encrypted (`v1:` via `encryptField`); null until ready |
| `created_at` | timestamptz NOT NULL default now() | |

**RLS:** owner-only `SELECT`; writes via service-role only. Entitlement = a row exists; deliverable = `status='ready'` with a decryptable `report`.

Migration applied via Supabase MCP; `get_advisors` must be clean (intended `rls_enabled_no_policy`-free since these have owner SELECT policies; service-role writes bypass RLS).

### `lib/credits/` (pure, server-usable)
- `creditBalance(rows): number` = `sum(delta)` (integer math only, never float).
- `outstandingLiability(rows): number` = `Σ(remaining credits × unit_price_cents)` over live grants — the basis for the future `/admin/liability` view (built later; the helper lands now).
- Store helpers in `lib/credits/store.ts`: `getLedger(client, userId)`, `grantCredits(admin, {...})` (service-role insert, idempotent on `stripe_event_id`), `getWellnessProfile(client, userId, sessionId)`, `upsertEntitlement(admin, {...})`, `saveReport(admin, id, encryptedReport)`.

### `lib/stripe/` 
- `lib/stripe/server.ts`: lazily-constructed Stripe client from `STRIPE_SECRET_KEY` (server-only). No hardcoded keys.
- Env (added to `.env.example`): `STRIPE_SECRET_KEY` (`sk_test_…`), `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` (`pk_test_…`), `STRIPE_WEBHOOK_SECRET` (`whsec_…`), `STRIPE_PRICE_WELLNESS_PROFILE` (`price_…`), `CREDIT_EXPIRY_MONTHS` (default 12), `WELLNESS_PROFILE_CREDITS` (default 12), `WELLNESS_PROFILE_PRICE_CENTS` (default 2900).

### Routes
- **`POST /api/checkout`** — authed + full geo/disclosure gate (mirrors `/api/chat`). Creates a Stripe Checkout Session (`mode: payment`, the one line item), `metadata: { userId, sessionId }`, `client_reference_id = userId`, `success_url=/profile?checkout=success`, `cancel_url=/home?checkout=cancelled`. Returns `{ url }`.
- **`POST /api/stripe/webhook`** — public route; trust = the signature. Reads the **raw body** (`await req.text()`), verifies `Stripe-Signature` against `STRIPE_WEBHOOK_SECRET` via `stripe.webhooks.constructEvent` (forged/unsigned → 400). Handles `checkout.session.completed` only (others → 200 ignore). Extracts `userId`/`sessionId` from metadata. Idempotently, via the service-role admin client: insert a `+WELLNESS_PROFILE_CREDITS` ledger row (`unit_price_cents = round(WELLNESS_PROFILE_PRICE_CENTS / WELLNESS_PROFILE_CREDITS)`, `expires_at`, `reason='purchase:wellness_profile'`, `stripe_event_id=event.id`) and upsert the `wellness_profiles` entitlement (`stripe_checkout_id=session.id`, `status='paid'`). Returns 200 fast. No report generation here (no transcript).
- **`POST /api/profile/generate { sessionId, messages }`** — authed + gated. (1) Verify a paid entitlement (`status in ('paid','ready')`); webhook not landed yet → `{ status: 'pending' }`. (2) Already `ready` → return ok, no regenerate (idempotent; no double Sonnet spend). (3) Else: Sonnet (dedicated profile-report prompt, distinct from the session summary) over the transcript → `scanOutput` (regenerate-once-then-fail, per the summarizer) → `encryptField` → `saveReport` + `status='ready'` → **discard the transcript** (never stored raw). Reuses `createSonnetCaller` + the summarizer's filter/encrypt discipline. An output-filter block logs a new `profile_filtered` compliance event (added to the `ComplianceEvent` union, mirroring `summary_filtered`).

### Page + components
- **`app/(app)/profile/page.tsx`** — server component, authed, RLS client; reads the user's latest `wellness_profiles` + `health_scores`; renders the ready/preparing/locked states. The "preparing" + checkout-return generation trigger is a small client island.
- **`lib/chat/decision.ts`** — pure: `DECISION_DELAY_TURNS = 2`; `shouldShowDecision(turnsSinceGap, alreadyShown, alreadyPurchased): boolean`.
- **`components/chat/DecisionCard.tsx`** — inline card (reuses `AvaBubble`); props `{ onBuy, onContinueFree, onShare, busy }`.
- **`lib/chat/checkout.ts`** — `createCheckout(sessionId?): Promise<{ ok:true, url }|{ ok:false }>` (never-throws browser wrapper, like `createShareCard`).
- **`MessageList`** union extended to `TextItem | GapItem | DecisionItem`; the existing `isTextItem` guard already excludes non-text items from the LLM wire.
- **`ChatScreen`** — tracks the turn index when the gap fired; fires the `DecisionItem` once at the +2-turn mark (unless `alreadyPurchased`/dismissed); resets on `endCheckIn`. `app/(app)/home/page.tsx` passes `alreadyPurchased` (from a server lookup of the user's entitlements).

### Data flow

```
Gap card shown (1D-a) → +2 user turns → DecisionItem appended → DecisionCard
  └─ "Unlock my profile" → createCheckout() → /api/checkout → Stripe-hosted page
       └─ pay (sandbox) → redirect /profile?checkout=success     ┌─ webhook (async) ─┐
            └─ /profile fires /api/profile/generate(transcript)   │ verify sig        │
                 └─ entitlement paid? → Sonnet → scanOutput →      │ grant 12 credits  │
                    encrypt → status=ready → discard transcript    │ upsert entitlement│
                      └─ /profile renders report + radar           └───────────────────┘
```

No card data on our servers; transcript discarded after generation; grants service-role-only + idempotent.

## Security & compliance

- **Webhook:** signature-verified (HMAC over raw body); idempotent on `stripe_event_id` (ledger) and `stripe_checkout_id` (entitlement) so retries/replays never double-grant; `userId` from server-set metadata, never the client.
- **Grants service-role-only:** RLS blocks all user writes to `credit_ledger`/`wellness_profiles`; a user can only read their own rows.
- **No secrets in code:** all Stripe keys + price id via env; `.env.example` updated.
- **Two artifacts never merged (rule #5):** the report is the private profile — encrypted, in-account, RLS-owner-only, with no code path to the share endpoint / brag card. The `/profile` route is gated; the share path (`/api/share`, `/share/[token]`) is untouched and cannot read `wellness_profiles`.
- **No medical claims (rule #2):** the report passes the same `scanOutput` output filter (regenerate-once-then-fail), non-diagnostic.
- **Data minimization (rule #4):** transcript used transiently for generation then discarded; only the filtered, encrypted report is stored.
- **Liability (rule #8):** every grant carries `unit_price_cents` + `expires_at` from day one; balance = `sum(delta)`; refund is *possible* by construction (the data to reverse exists). The expiry job / refund paths / admin reserve view are deferred (below) but the model supports them.

## Testing

All Stripe interaction is **mocked** in vitest (no account/keys needed to build or test).
- `lib/credits/` — `creditBalance`/`outstandingLiability` pure unit tests (integer math, expiry exclusion).
- `lib/chat/decision.ts` — trigger edge tests (delay, once-only, suppression).
- `DecisionCard` — render/click (buy/continue/share callbacks), no leak.
- `MessageList` — renders a decision item inline.
- `ChatScreen` — decision fires once at +2 turns; suppressed when `alreadyPurchased`; reset on end.
- `/api/checkout` — auth/gate, builds the session with correct metadata (Stripe mocked).
- `/api/stripe/webhook` — signature verify (reject forged), idempotent grant + entitlement (retry = no-op), ignores non-checkout events.
- `/api/profile/generate` — entitlement gate (pending/paid/ready), Sonnet mocked, output-filter path, encrypt + discard, idempotent.
- `/profile` page — ready/preparing/locked rendering.

## Decomposition (4 sub-slices — each its own plan + PR + security review)

- **1D-b — Credit ledger foundation.** Migration `0007` + `lib/credits/` (helpers + store). No Stripe, no UI.
- **1D-c — Stripe checkout + webhook.** `lib/stripe/`, `POST /api/checkout`, `POST /api/stripe/webhook` (sig-verified, idempotent grant + entitlement). Stripe mocked.
- **1D-d — Report + `/profile` page.** `POST /api/profile/generate` + the `/profile` page.
- **1D-e — Inline Decision card + chat wiring.** `lib/chat/decision.ts`, `DecisionCard`, `MessageList` union, `ChatScreen`, `createCheckout`, return handling.

Build order b → c → (d, e). `writing-plans` produces a plan per slice, starting with 1D-b.

## Explicitly deferred (NOT in this feature)

- Credit **expiry job** (field exists; daily `expire:ttl` writer is later).
- **Refund / chargeback / session-failure** paths (atomic Stripe-reverse + credit removal; dispute freeze).
- **`/admin/liability`** reserve view, **`APP_SUNSET_MODE`**, **bulk refund** — exit switches, pre-GATE-1.
- **Subscriptions** (Plus/Pro/Max).
- **Starter $9 / Deep $49** products.
- Credits being **spent** (avatar metering = 1F).
- Server-side **background report generation** if the tab closes before generate fires (test-mode acceptable without).
- **Live mode** (`sk_live`) — GATE-1 (attorney sign-off, incl. credit-expiration legality).
